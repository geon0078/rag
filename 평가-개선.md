| 산출물 | 위치 | 용도 |
|---|---|---|

### 1.2 이 단계의 목표

1. 우리 데이터에 최적인 RAG 파이프라인 조합 자동 발견
2. 정량적 품질 지표 확보 (recall, faithfulness 등)
3. 운영 배포 전 회귀 테스트 기준선 확립

### 1.3 통과 기준 (이 값을 못 넘으면 다음 단계로 가지 말 것)

| 지표 | 목표 | 의미 |
|---|---|---|
| `retrieval_recall@5` | ≥ 0.90 | 정답 청크가 top-5에 포함될 확률 |
| `g_eval` (faithfulness) | ≥ 0.85 | 답변이 컨텍스트에 충실 |
| `RAGAS faithfulness` | ≥ 0.85 | 환각 검증 통과 |
| Negative 거절률 | ≥ 0.80 | 정답 없는 질문에 거절 답변 |
| 카테고리 라우팅 정확도 | ≥ 0.95 | 컬렉션 분류 정확성 |
| 캠퍼스 필터 정확도 | = 1.00 | 100% 필요 (잘못 답하면 사고) |

### 2.2 질문 유형 분포

| 유형 | 비율 | 예시 |
|---|---|---|
| **Single-hop** | 70% (115개) | "졸업 학점은?" → 1개 청크로 답변 |
| **Multi-hop** | 20% (33개) | "성남캠퍼스 간호학과 졸업 요건" → 학과정보 + 학칙 |

### 2.3 qa.parquet 스키마 (AutoRAG 표준)

| 컬럼 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `qid` | str | ✓ | QA 고유 ID (예: `qa_001`) |
| `query` | str | ✓ | 사용자 질문 |
| `retrieval_gt` | list[list[str]] | ✓ | 정답 청크 ID 목록 (이중 리스트) |
| `generation_gt` | list[str] | ✓ | 정답 답변 (1개 이상의 허용 답변) |
| `qa_type` | str | ✓ | single_hop / multi_hop / filter_required / negative |
| `collection` | str | ✓ | 정답이 속한 컬렉션 |

### 3.7 청크 샘플링 전략

```python
def sample_chunks_for_qa(corpus_df, target_per_collection):
    """컬렉션별 청크 샘플링 - 다양성 확보"""
    samples = []
    for collection, target in target_per_collection.items():
        col_df = corpus_df[corpus_df["source_collection"] == collection]

        # 강의평가: 같은 강의에서 1개만
        if collection == "강의평가":
            col_df["lecture_id"] = col_df["chunk_id"].str.extract(r"lec_lecture_reviews_(\d+)_")
            sampled = col_df.groupby("lecture_id").apply(
                lambda g: g.sample(1, random_state=42)
            ).reset_index(drop=True)
            sampled = sampled.sample(min(target, len(sampled)), random_state=42)
        else:
            # 너무 짧은 청크 제외
            col_df = col_df[col_df["chunk_chars"] >= 50]
            sampled = col_df.sample(min(target, len(col_df)), random_state=42)

        samples.append(sampled)

    return pd.concat(samples, ignore_index=True)
```

## 4. AutoRAG 평가 실행 명세

### 4.1 평가 설정 파일

기존 `configs/autorag.yaml`에 추가/수정해야 할 부분:

```yaml
# AutoRAG 평가 메트릭 설정
node_lines:
  - node_line_name: retrieve_node_line
    nodes:
      - node_type: retrieval
        strategy:
          metrics:
            - retrieval_f1
            - retrieval_recall
            - retrieval_precision
            - retrieval_ndcg
            - retrieval_map
            - retrieval_mrr
        # ★ Phase 5에서는 모든 조합을 평가
        top_k: [3, 5, 10]
        modules:
          - module_type: bm25
            bm25_tokenizer: [ko_okt, ko_kkma, ko_kiwi]
          - module_type: vectordb
            vectordb: solar_embed
          - module_type: hybrid_rrf
            target_modules: [bm25, vectordb]
            rrf_k: [40, 60, 80]

      - node_type: passage_reranker
        strategy:
          metrics: [retrieval_f1, retrieval_recall, retrieval_ndcg, retrieval_mrr]
        top_k: [3, 5]
        modules:
          - module_type: pass_reranker  # 리랭커 없음 비교
          - module_type: upstage_reranker  # API
          - module_type: flag_embedding_reranker  # 자체 호스팅
            model_name: dragonkue/bge-reranker-v2-m3-ko
            batch: 8

  - node_line_name: post_retrieve_node_line
    nodes:
      - node_type: prompt_maker
        strategy:
          metrics:
            - metric_name: bleu
            - metric_name: rouge
            - metric_name: meteor
            - metric_name: sem_score
              embedding_model: upstage_embed
            - metric_name: g_eval
              metrics: [coherence, consistency, fluency, relevance]
              model: solar-pro
        modules:
          - module_type: fstring
            prompt: ["{우리_프롬프트_템플릿}", "{대안_프롬프트}"]

      - node_type: generator
        strategy:
          metrics:
            - metric_name: bleu
            - metric_name: rouge
            - metric_name: sem_score
              embedding_model: upstage_embed
            - metric_name: g_eval
              metrics: [coherence, consistency, fluency, relevance]
              model: solar-pro
        modules:
          - module_type: openai_llm
            llm: openai
            model: [solar-pro, solar-mini]
            api_base: https://api.upstage.ai/v1/solar
            temperature: [0.0, 0.3]
            batch: 4
```

### 4.3 실행 시간 예상

| 단계 | 예상 시간 | 비고 |
|---|---|---|
| 검색 모듈 평가 (BM25 3종 × Top-K 3종) | ~30분 | 165개 QA × 9개 조합 |
| 검색 융합 (RRF k 3종) | ~15분 | |
| 리랭커 평가 (3종 × top-K 2종) | ~1시간 | bge-reranker가 가장 오래 걸림 |
| Prompt 비교 | ~30분 | LLM 호출 발생 |

| **총** | **~4~5시간** | API rate limit 시 더 길어짐 |

## 5. RAGAS 보조 평가 명세

### 5.1 왜 RAGAS도 함께 사용하나

AutoRAG의 `g_eval`과 RAGAS의 메트릭은 측정 각도가 다릅니다.

| 도구 | 강점 |
|---|---|
| AutoRAG g_eval | 4축(coherence·consistency·fluency·relevance) 평가, 자동 최적화 |
| RAGAS | 산업 표준 RAG 전용 메트릭, faithfulness가 가장 신뢰도 높음 |

두 도구 점수가 **모두** 통과해야 안전합니다.

### 5.2 파일 위치
`scripts/evaluate_ragas.py`

### 5.3 평가 메트릭 (5가지)

```python
from ragas.metrics import (
    faithfulness,           # ★ 답변이 컨텍스트에 충실 (환각 검증)
    answer_relevancy,       # 답변이 질문에 직접 답변
    context_precision,      # 검색된 컨텍스트가 질문과 관련
    context_recall,         # 정답에 필요한 정보가 컨텍스트에 있음
    context_entity_recall,  # 정답의 핵심 개체가 포함됨
)
```

### 5.4 실행 흐름

```python
# 의사코드
1. data/qa.parquet 로드
2. 각 QA에 대해:
   a. RAG 파이프라인 실행 → answer + retrieved_contexts 획득
   b. RAGAS dataset 형식으로 변환:
      {
        "question": query,
        "answer": rag_answer,
        "contexts": retrieved_contexts,
        "ground_truth": generation_gt[0]
      }
3. RAGAS evaluate() 실행 (Solar 모델 사용)
4. 결과를 reports/ragas_report.json 저장
5. 카테고리별 점수 분포 분석 → reports/ragas_by_collection.csv
```

### 5.5 코드 구조

```python
# scripts/evaluate_ragas.py
import pandas as pd
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall
from langchain_upstage import ChatUpstage, UpstageEmbeddings
from datasets import Dataset
from src.pipeline.rag_pipeline import rag_query  # Phase 4에서 만든 파이프라인

async def run_ragas_evaluation():
    qa_df = pd.read_parquet("data/qa.parquet")

    # negative 케이스 제외 (RAGAS는 정답 없는 케이스 평가 어려움)
    qa_df = qa_df[qa_df["qa_type"] != "negative"]

    samples = []
    for _, row in qa_df.iterrows():
        result = await rag_query(row["query"])
        samples.append({
            "question": row["query"],
            "answer": result["answer"],
            "contexts": [c["contents"] for c in result["contexts"]],
            "ground_truth": row["generation_gt"][0],
        })

    dataset = Dataset.from_list(samples)

    score = evaluate(
        dataset,
        metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
        llm=ChatUpstage(model="solar-pro"),
        embeddings=UpstageEmbeddings(model="solar-embedding-1-large-query"),
    )

    # 저장
    score.to_pandas().to_json("reports/ragas_report.json", orient="records")
    print(f"Faithfulness: {score['faithfulness']:.3f}")
    print(f"Answer Relevancy: {score['answer_relevancy']:.3f}")
    print(f"Context Precision: {score['context_precision']:.3f}")
    print(f"Context Recall: {score['context_recall']:.3f}")
```

### 6.1 Negative 거절률 평가

`scripts/eval_negative.py`:

```python
async def evaluate_negative():
    qa_df = pd.read_parquet("data/qa.parquet")
    negative = qa_df[qa_df["qa_type"] == "negative"]

    rejected = 0
    for _, row in negative.iterrows():
        result = await rag_query(row["query"])
        # 거절 키워드 포함 여부
        is_rejected = any(kw in result["answer"] for kw in [
            "찾을 수 없", "제공된 자료", "확인할 수 없", "정보가 없"
        ])
        # 또는 grounded == False
        if is_rejected or not result["grounded"]:
            rejected += 1

    rejection_rate = rejected / len(negative)
    print(f"Negative 거절률: {rejection_rate:.2%}")
    assert rejection_rate >= 0.80, "거절률 미달"
```

```python
async def evaluate_campus_filter():
    qa_df = pd.read_parquet("data/qa.parquet")
    filter_qa = qa_df[qa_df["qa_type"] == "filter_required"]

    correct = 0
    for _, row in filter_qa.iterrows():
        expected_campus = row["metadata"]["campus_filter"]  # 성남 / 의정부
        result = await rag_query(row["query"])

        # 검색된 모든 청크의 campus가 일치하거나 '전체'인지 확인
        all_match = all(
            c["metadata"]["campus"] in [expected_campus, "전체"]
            for c in result["contexts"]
        )
        if all_match:
            correct += 1

    accuracy = correct / len(filter_qa)
    print(f"캠퍼스 필터 정확도: {accuracy:.2%}")
    assert accuracy == 1.0, "캠퍼스 필터 100% 정확도 미달 - 즉시 수정 필요"
```

```python
async def evaluate_routing():
```

## 11. 자주 발생하는 문제와 해결

### 문제 1: AutoRAG 평가 중 Solar API rate limit
- **증상**: HTTP 429 에러 다발
- **해결**:
  - `batch` 크기를 4 → 2로 축소
  - 평가를 컴포넌트별로 분리 실행 (`autorag evaluate --node retrieval` 단계적)
  - 야간 시간대 실행

### 문제 2: 자동 생성 QA 품질 낮음
- **증상**: 같은 질문 반복, 답변이 모호
- **해결**:
  - 프롬프트에 "다양한 학년 학생이 물을 만한 질문" 명시
  - `temperature: 0.7`로 다양성 확보
  - 청크 샘플링 시 같은 강의/조항 중복 방지

### 문제 3: 강의평가 RAGAS faithfulness 낮음
- **증상**: 강의평가만 0.7 미만
- **해석**: 강의평가는 "주관적 의견"이라 ground_truth 매칭이 어려움
- **해결**:
  - 강의평가 QA의 `generation_gt`를 의미 매칭 위주로 작성
  - g_eval과 sem_score 위주로 평가, RAGAS faithfulness는 보조 지표로

### 문제 4: 한자/특수문자 포함 질문 실패
- **증상**: "한자(漢字)" 같은 질문에서 검색 실패
- **해결**: BM25 토크나이저 ko_okt 사용 (한자 처리 가능)
