## 1. 이전 진단의 정확성 검토

### 1.1 이전 답변에서 맞은 점

다음 진단은 산업 표준과 일치합니다:

- ✅ **Trial F (paraphrase) 97%는 신뢰할 수 없는 숫자**라는 판단은 정확
- ✅ **Adversarial 53.6%가 실제 시스템 성능에 더 가까움**이라는 통찰도 정확
- ✅ **paraphrase test set bias**는 Anyscale·RAGAS·Google Cloud 모두 인정하는 함정
- ✅ Mode E·C 즉시 fix 우선순위 자체는 합리적

### 1.2 이전 답변에서 부족했던 점

다음은 더 정확하게 다뤄야 합니다:

- ❌ "Adversarial OK rate ≥ 75% 통과 기준"은 임의로 정한 숫자였음 — 산업 표준이 없음, 도메인별 보정 필요
- ❌ **Synthetic vs Adversarial 이분법**으로만 설명했지만, 실제로는 **3-tier 평가셋 구조**가 표준 (Golden / Synthetic / Production-sourced)
- ❌ **인간 평가(human-in-the-loop)와 LLM judge 보정** 절차를 빼먹음 — 이게 더 근본적
- ❌ Mode E `notSure` 통과 정책은 답변 환각 위험 존재 — **claim-level faithfulness 검증** 같은 보조 안전장치 필요
- ❌ **계층별 분리 평가** (retrieval ≠ generation) 원칙을 강조 못 함

### 1.3 이번 명세서의 개선 방향

산업 표준에 따라 4가지를 새로 도입합니다:

1. **3-tier 평가셋 구조** (Golden / Synthetic / Adversarial)
2. **Retrieval과 Generation 분리 측정**
3. **Claim-level faithfulness** (RAGChecker 패턴)
4. **Human-LLM judge 보정** (calibration)

## 2. 왜 paraphrase test set은 신뢰할 수 없나

### 2.1 paraphrase bias의 메커니즘

자동 생성 QA의 일반적 흐름:

```
청크 본문: "졸업을 위해서는 최소 130학점 이상을 취득하여야 한다"
         ↓
LLM이 질문 생성: "졸업을 위해 취득해야 하는 최소 학점은?"
         ↓
정답: "130학점"
```

문제는 **질문이 청크 본문의 어휘를 그대로 사용**한다는 점입니다. 이러면:
- 임베딩 검색이 거의 100% 정답 청크를 찾음 (당연함, 같은 단어들)
- BM25도 키워드가 일치하니까 잘 찾음
- 결과적으로 검색 정확도 97%가 나옴

하지만 실제 학생은 이렇게 묻습니다:
```
"졸업학점 몇 개야?"  (학점·졸업 → 본문에 없음)
"졸업하려면 몇 학점 이수해야 함?"  (이수 → 본문 표현 다름)
"학점 몇 개 따야 졸업?"  (어순 도치, 구어)
```

**같은 의도, 다른 어휘** → 임베딩 매칭 정확도 급락 → 53.6%.

### 2.2 산업 표준이 제시하는 해법

여러 출처에서 일관되게 권고하는 패턴 ([Anyscale RAG eval](https://docs.anyscale.com/rag/evaluation), [Google Cloud RAG best practices](https://cloud.google.com/blog/products/ai-machine-learning/optimizing-rag-retrieval), [labelyourdata.com](https://labelyourdata.com/articles/llm-fine-tuning/rag-evaluation), [premai.io](https://blog.premai.io/rag-evaluation-metrics-frameworks-testing-2026/)):

| 평가셋 종류 | 용도 | 만드는 법 |
|---|---|---|
| **Golden** | 정확한 메트릭 보정 (50~200건) | 사람이 수동으로 작성, 정답 검증됨 |
| **Synthetic** | 회귀 테스트 (500~1000건) | LLM 자동 생성, paraphrase 위주 |
| **Production-sourced** | 실 사용자 만족도 (지속 누적) | 실제 학생 로그에서 샘플링 |

핵심 원칙: **"Synthetic 점수 ≥ Golden 점수 ≥ Production 점수"** 가 일반적이고, **세 점수의 갭이 클수록 평가 편향이 크다**는 신호입니다.

우리 경우:
- Synthetic (Trial F) 97%
- Adversarial(Golden 근사) 53.6%
- 갭 +43pt → **평가 편향이 매우 큼**

### 2.3 중요한 통찰

> "High scores on public benchmarks don't guarantee good performance on domain-specific, production data." — Anyscale RAG eval docs

**우리 시스템은 이미 운영 진입에 적합하지 않은 상태**입니다. Synthetic 97%는 자기기만이고, 진짜 성능은 53.6%입니다. 이 갭을 줄이는 것이 다음 작업의 본질입니다.

## 3. 3-Tier 평가셋 구조 (산업 표준)

### 3.1 Tier 1 — Golden Set (가장 중요)

**목적**: 정확한 메트릭 보정. LLM judge의 신뢰도를 사람 판단에 맞춤.

| 속성 | 값 |
|---|---|
| 규모 | 50~200건 (우리는 100건 권장) |
| 작성자 | 사람 (학생지원팀 + 개발자) |
| 검증 | 정답이 corpus에 실제 있는지 사람이 확인 |
| 어휘 | 학생 자연 표현, paraphrase 금지 |
| 변경 빈도 | 분기 1회 (frozen) |
| 저장 | git에 커밋, 버전 관리 |

#### Golden Set 작성 가이드

학생이 던질 만한 자연스러운 질문 100건을 다음 패턴으로 균형 있게:

| 패턴 | 비중 | 예시 |
|---|---|---|
| 일반 질의 (반말·구어) | 25% | "학식 메뉴 뭐야?" |
| 정식 표현 | 20% | "교내 도서관 운영시간을 알려주세요" |
| 축약·구어 | 15% | "수강신청 언제?" |
| 오타·비정형 | 10% | "졸업학첨", "수가신청" |
| 복합 질문 (multi-hop) | 10% | "성남캠 도서관 24시간? 시험기간엔?" |
| 캠퍼스 명시 | 10% | "의정부캠 학식 메뉴" |
| 정답 없는 질문 (negative) | 10% | "오늘 날씨", "교수님 연락처" |

각 항목에는:
- `query`: 학생 표현 그대로
- `intent_category`: 의도 분류 (학칙·일정·FAQ·시설·negative)
- `expected_doc_ids`: 정답 청크들 (사람이 corpus 보고 직접 선택)
- `expected_answer`: 모범 답변 (사람 작성, 1~3문장)
- `expected_grounded`: 답이 가능한가 (true/false, negative는 false)
- `difficulty`: easy/medium/hard
- `source`: 어떻게 만들었나 (직접 작성·학생 로그·기존 set)

### 3.2 Tier 2 — Synthetic Set (회귀 테스트용)

**목적**: 코드·인덱싱·청킹이 깨지지 않았는지 빠르게 회귀 감지.

| 속성 | 값 |
|---|---|
| 규모 | 500건 (우리 기존 qa.parquet) |
| 작성자 | LLM 자동 생성 |
| 검증 | 사람이 일부 샘플(10%)만 검수 |
| 어휘 | 청크 본문 paraphrase |
| 변경 빈도 | 인덱싱 변경 시 재생성 |
| 저장 | parquet, 갱신 가능 |

**핵심**: Synthetic Set은 통과 기준이 아니라 **회귀 감지 신호**입니다. 기존에 통과하던 케이스가 갑자기 실패하면 코드 문제. 점수의 절대값(97%)은 의미 없음.

### 3.3 Tier 3 — Production-sourced Set (운영 진입 후)

**목적**: 실제 사용자 만족도 추정.

| 속성 | 값 |
|---|---|
| 규모 | 운영 누적 (목표: 6개월에 200건) |
| 작성자 | 실제 학생 로그에서 샘플링 |
| 검증 | 운영자가 라벨링 |
| 어휘 | 실 사용자 자연 표현 |
| 변경 빈도 | 매주 추가 |
| 저장 | DB, 익명화 |

**개발 단계인 지금은 비어있음.** 운영 진입 시점부터 매주 누적.

### 3.4 평가셋 갱신 정책

| 셋 | 사용 시점 | 통과 기준 |
|---|---|---|
| Golden Set | 모든 평가 (변경마다) | **이게 진짜 통과 기준** |
| Synthetic Set | 매 빌드 (CI) | 점수 자체가 아닌 회귀(prev → curr 감소)만 감지 |
| Production Set | 운영 진입 후 매주 | 운영 KPI 보조 신호 |

## 4. Retrieval과 Generation 분리 측정

이전 평가는 end-to-end OK rate만 봤지만, **단계별 분리 측정**이 산업 표준입니다 ([toloka.ai 가이드](https://toloka.ai/blog/rag-evaluation-a-technical-guide-to-measuring-retrieval-augmented-generation/), [N-iX best practices](https://www.n-ix.com/rag-evaluation/)).

### 4.1 왜 분리해야 하나

end-to-end만 보면 **어디서 실패했는지 알 수 없습니다**:

```
질문 → [검색 실패] → 잘못된 청크 → [생성은 정상] → 잘못된 답변
질문 → [검색 정상] → 정답 청크   → [생성 실패] → 잘못된 답변
```

두 케이스 모두 OK rate에서 fail로 카운트되지만, 원인이 다릅니다. 둘을 분리해야 어디를 고칠지 알 수 있습니다.

### 4.2 Retrieval 메트릭 (정답 청크 검색 정확도)

Golden Set의 `expected_doc_ids`를 사용해서:

| 메트릭 | 의미 | 목표 |
|---|---|---|
| `recall@5` | 정답 청크가 top-5에 포함된 비율 | ≥ 0.85 |
| `recall@10` | top-10 포함 비율 | ≥ 0.95 |
| `MRR` | 첫 정답의 역순위 평균 | ≥ 0.65 |
| `nDCG@5` | 순위 가중 정확도 | ≥ 0.75 |

각 항목을 **카테고리별로도 측정** (FAQ만, 학칙만, 강의평가만 등) — 어느 컬렉션이 약한지 파악.

### 4.3 Generation 메트릭 (답변 품질)

답변이 검색 결과를 잘 활용했는지:

| 메트릭 | 의미 | 측정 방법 |
|---|---|---|
| `faithfulness` | 답변의 모든 주장이 컨텍스트에 근거 | RAGAS or RAGChecker (claim-level) |
| `answer_relevance` | 답변이 질문에 직접 답함 | RAGAS LLM judge |
| `correctness` | 답변이 expected_answer와 일치 | LLM judge (Golden Set만) |
| `citation_accuracy` | 인용한 청크가 실제 사용된 정보 출처 | 후처리 패턴 매칭 |

### 4.4 분리 측정으로 알 수 있는 진단

| recall | faithfulness | 진단 | 조치 |
|---|---|---|---|
| 낮음 | 낮음 | 검색 실패 → 생성 실패 (연쇄) | **검색 먼저 고치기** |
| 낮음 | 높음 | 검색 실패해도 거절 답변 잘 함 | 검색 보강 (HyDE 강화·메타 필터·청킹) |
| 높음 | 낮음 | 검색은 잘 하는데 생성이 환각 | 프롬프트 개선·LLM 변경·Groundedness 강화 |
| 높음 | 높음 | 정상 | 다른 메트릭 점검 |

**우리 케이스**: Adversarial 53.6%인데 retrieval/generation 분리 안 됐으니 일단 분리 측정부터 해야 합니다.

## 5. Claim-level Faithfulness (RAGChecker 패턴)

### 5.1 일반 faithfulness의 한계

RAGAS의 표준 faithfulness는 답변 전체가 grounded인가만 판단합니다. 하지만 답변이 5문장이면 그중 1문장만 환각이어도 사고가 됩니다.

> "RAGChecker decomposes answers into individual statements and evaluates each against the retrieved context using entailment checks." — N-iX

### 5.2 Claim-level 검증 방식

답변을 atomic claim 단위로 쪼개서 각각 검증:

```
답변: "을지대 졸업학점은 130학점이며, 의학과는 별도 규정이고,
      매 학기 15학점 이상 이수해야 합니다."

       ↓ atomic claim 분해

[Claim 1] 을지대 졸업학점은 130학점
[Claim 2] 의학과는 별도 규정
[Claim 3] 매 학기 15학점 이상 이수해야 함

       ↓ 각각 컨텍스트와 entailment 검증

[Claim 1] supported (학칙_제31조)
[Claim 2] supported (학칙_제31조 3항)
[Claim 3] NOT_SUPPORTED ← 환각!

faithfulness = 2/3 = 0.67
```

### 5.3 우리 시스템 적용 방법

Solar Pro로 두 단계 호출:

| 단계 | 호출 | 출력 |
|---|---|---|
| 1. Claim 추출 | "다음 답변을 사실 단위로 쪼개서 list로 출력" | atomic claims |
| 2. Entailment 검증 | 각 claim이 컨텍스트에서 추론 가능한가 | supported/contradicted/not_in_context |

UpstageGroundednessCheck는 답변 전체에 단일 라벨을 주지만, RAGChecker 패턴은 **부분 환각도 잡습니다.** 우리 시스템에 추가하면 Mode E·C 이슈가 더 정밀하게 감지됩니다.

### 5.4 Mode E 완화 정책의 안전장치

이전 답변에서 `notSure`도 통과시키자고 제안했는데, 이건 환각 위험이 있습니다. **Claim-level faithfulness가 보조 안전장치**가 됩니다:

```
1차: Groundedness Check (전체) → grounded / notSure / notGrounded
2차: Claim-level faithfulness 점수
3차 정책:
  - grounded → 통과
  - notSure + faithfulness ≥ 0.8 → 통과 (multi-hop 케이스 회복)
  - notSure + faithfulness < 0.8 → 차단 (환각 의심)
  - notGrounded → 차단
```

이러면 Mode E를 회복하면서도 환각이 늘지 않습니다.

## 6. LLM Judge 보정 (Calibration)

### 6.1 LLM judge의 알려진 편향

> "LLM judges exhibit systematic biases—preferring longer responses, showing positional bias, and favoring their own outputs." — Anyscale

LLM judge가 평가하는 점수는 사람 판단과 다를 수 있습니다. 특히:
- **길이 편향**: 긴 답변을 더 정확하다고 판단
- **자기 선호**: 같은 모델이 만든 답변을 더 좋게 봄
- **위치 편향**: 컨텍스트 앞쪽 정보를 더 중요시

### 6.2 보정 절차 (Inter-Annotator Agreement)

표준 방법 ([Maxim AI 가이드](https://www.getmaxim.ai/articles/building-a-golden-dataset-for-ai-evaluation-a-step-by-step-guide/)):

1. Golden Set 100건에 대해 사람이 직접 OK/Fail 라벨 (운영자 1명)
2. 같은 100건을 LLM judge로 평가
3. **agreement rate** 계산 (사람과 LLM이 같은 라벨을 준 비율)
4. agreement < 80%면 LLM judge 프롬프트 개선
5. agreement ≥ 80%면 LLM judge 점수 신뢰

**우리 시스템에서는** RAGAS faithfulness가 사람 판단과 얼마나 일치하는지 측정한 적이 없습니다. 적어도 한 번은 보정해야 합니다.

### 6.3 간이 보정 (개발 단계)

전체 보정이 부담되면 **20~30건**만 사람 라벨링해서 LLM judge와 비교:
- agreement ≥ 0.8 → judge 신뢰
- agreement < 0.8 → judge 프롬프트 수정 후 재시도

## 7. 즉시 fix할 실패 모드 (Mode E·C·A 재해석)

### 7.1 Mode E (23.2%) — Groundedness 거짓 음성

**진단 재확인**: 정답 청크가 검색되었는데도 Groundedness가 `notSure`/`notGrounded`로 거부 → contexts:[] fallback.

**원인**: paraphrase가 아닌 자유 질문은 답변이 컨텍스트와 어휘가 다름 → Groundedness가 보수적으로 판단.

**fix 방향**:
1. Claim-level faithfulness 도입 (Section 5)
2. notSure + faithfulness ≥ 0.8이면 통과 + "자료를 종합한 답변" 워닝
3. notGrounded는 여전히 차단

**검증 방법**: Golden Set의 `expected_grounded=true` 케이스 중 거부된 것의 비율이 11.7% → ≤5% 목표.

### 7.2 Mode C (13.2%) — 잘못된 청크 인용

**진단 재확인**: top-1 청크가 정답인데, LLM이 top-2~5의 정보를 답변에 섞음.

**원인**: 컨텍스트 포맷이 청크 경계 명시 안 함 → LLM이 어느 청크가 1위인지 모름.

**fix 방향**:
1. 컨텍스트에 [문서 1], [문서 2] 명시 + 점수 표시
2. 시스템 프롬프트에 "top-1 우선 사용. 부족할 때만 top-2~5" 명시
3. 답변 후처리로 인용 형식 강제

**검증 방법**: Golden Set의 `expected_doc_ids[0]`이 답변 인용에 포함된 비율 측정. 90% 이상 목표.

### 7.3 Mode A (10%) — 강의평가 검색 정확도

**진단 재확인**: 강의평가 카테고리에서 reranker 정확도 낮음.

**원인 추정**: 강의평가는 의미 검색(Dense)이 더 중요한데, 현재 RRF 가중치가 균등.

**fix 방향**:
- **다음 sprint로 보류** (지금 단계에서 우선순위 낮음)
- 카테고리별 동적 가중치는 구조 변경 큼
- 우선 retrieval 분리 측정으로 진짜 문제 확인 후 결정

### 7.4 Mode E·C fix 우선순위

| 작업 | 효과 | 비용 |
|---|---|---|
| 1. Claim-level faithfulness 도입 | 매우 큼 (Mode E 회복 + 환각 방지) | 2시간 |
| 2. 컨텍스트 포맷 + top-1 우선 룰 | 큼 (Mode C 회복) | 1시간 |
| 3. Golden Set 100건 작성 | **가장 큼 (평가 신뢰도 회복)** | 8시간 (사람 작업) |
| 4. 분리 측정 도입 (retrieval/generation) | 큼 (진단 정확도) | 2시간 |
| 5. LLM judge 보정 | 보통 | 4시간 |

**총 약 17시간 = 2~3일 작업.**

## 8. 새 통과 기준 정의

### 8.1 통과 기준 표 (Golden Set 기준)

기존 통과 기준은 Trial F(synthetic) 점수였습니다. 이제 **Golden Set 점수로 변경**:

| 메트릭 | 측정 셋 | 임계값 | 의미 |
|---|---|---|---|
| `retrieval_recall@5` | Golden | ≥ 0.85 | 정답 청크 검색 |
| `retrieval_mrr` | Golden | ≥ 0.65 | 정답이 상위에 |
| `faithfulness` (claim-level) | Golden | ≥ 0.85 | 환각 없음 |
| `answer_correctness` | Golden | ≥ 0.80 | 답변 정확성 (LLM judge) |
| `citation_accuracy` | Golden | ≥ 0.90 | 인용 형식 |
| `negative_rejection_rate` | Golden | ≥ 0.80 | 정답 없는 질문 거절 |
| `category_routing@3` | Golden | ≥ 0.95 | 라우팅 정확도 |
| `campus_filter_accuracy` | Golden | = 1.00 | 캠퍼스 필터 |

### 8.2 Synthetic Set은 회귀만 감지

Synthetic Set의 점수는 **절대값을 사용하지 않습니다.** 대신:
- 이전 빌드 대비 ≥ -3pt 감소만 허용 (회귀 감지)
- 절대값이 80%든 95%든 의미 없음 (paraphrase bias)

### 8.3 갭 모니터링

평가 리포트에 **반드시 두 점수 병기**:

```
recall@5
  Golden:    0.87 ←  진짜 점수
  Synthetic: 0.96
  Gap:       +9pt  (acceptable, < 15pt)

faithfulness
  Golden:    0.78 ✗ (목표 0.85 미달)
  Synthetic: 0.94
  Gap:       +16pt ⚠️ (large, paraphrase bias 의심)
```

**갭이 15pt 이상이면 평가 신뢰도 낮음 신호** — Golden Set 보강 필요.

## 10. Golden Set 작성 디테일 (가장 중요)

### 10.1 작성 원칙

1. **paraphrase 금지** — 청크 본문 단어를 그대로 쓰지 말 것
2. **학생 자연 표현** — 반말·축약·구어 포함
3. **정답 검증** — 사람이 직접 corpus 열어서 정답 청크 선택
4. **다양성** — 7개 패턴 균형, 한 컬렉션에 몰리지 않게
5. **의도 다양성** — 정보 조회·절차 문의·비교 요청 모두 포함

### 10.2 작성 책임

| 단계 | 누가 | 무엇을 |
|---|---|---|
| 1차 작성 | 개발자 + Solar Pro 보조 | 후보 질문 150건 생성 |
| 2차 검증 | 개발자 1명 | corpus 보고 정답 청크 선택, 50건 채택 |
| 3차 보강 | 학생지원팀 1명 (가능하면) | 자연스러운 학생 표현 50건 추가 |
| 4차 frozen | 개발자 | git 커밋, v1.0 태그 |

### 10.3 100건 컬렉션 분포 권장

| 컬렉션 | 개수 | 이유 |
|---|---|---|
| 학칙_조항 | 15 | 정확성 중요 |
| FAQ | 20 | 가장 자주 사용 |
| 학사정보 | 15 | 절차 문의 많음 |
| 학사일정 | 10 | 시간 필터 검증 |
| 강의평가 | 15 | 가장 큰 컬렉션 |
| 시설_연락처 | 10 | 정확 매칭 |
| 장학금 | 5 | 도메인 특수 |
| 학과정보·교육과정 | 3 | 보조 |
| Negative | 7 | 거절 검증 |
| **합계** | **100** | |

### 10.4 자동 생성 시작점 (효율적)

전부 손으로 쓰지 말고, Solar Pro로 **다양한 표현의 후보**를 만들어서 사람이 선별:

```
Solar Pro 프롬프트:
"다음 corpus 청크를 보고 학생이 실제로 던질 만한 질문 5개를 만드세요.
규칙:
1. 각 질문은 표현 스타일이 달라야 함 (정중·반말·축약·구어·오타)
2. 청크 본문의 단어를 그대로 쓰지 말고, 학생이 자연스럽게 쓸 어휘로
3. 한 개는 의도가 모호한 짧은 질문 (예: '그거 어떻게 해?')

청크: {청크 내용}

출력: JSON list[str]"
```

5건 × 100청크 = 500건 후보에서 사람이 100건 선별. 이 방식이 표준입니다 ([Maxim AI 가이드](https://www.getmaxim.ai/articles/building-a-golden-dataset-for-ai-evaluation-a-step-by-step-guide/)).

## 12. 전체 평가 파이프라인 흐름

```
[코드 변경 / 인덱싱 변경]
        ↓
[Synthetic Set 회귀 측정]  ← 빠름, CI에서 매번
        ↓
[이전 대비 -3pt 이상 감소?]
        ├─ Yes → 즉시 알림, 변경 차단
        └─ No → 다음 단계
        ↓
[Golden Set 전체 측정] ← 정밀, 변경마다
        ├─ retrieval recall@5, MRR
        ├─ faithfulness (claim-level)
        ├─ answer_correctness (LLM judge)
        ├─ negative rejection
        ├─ category routing
        └─ campus filter
        ↓
[갭 분석: Golden vs Synthetic]
        ├─ 갭 ≥ 15pt → ⚠️ 평가 편향 의심, Golden 보강 필요
        └─ 갭 < 15pt → 정상
        ↓
[모든 메트릭 통과?]
        ├─ Yes → 다음 단계 (운영 진입 가능)
        └─ No → 메트릭별 디버그 가이드 (Section 4.4)
        ↓
[운영 진입 후]
        ↓
[Production Set 누적] ← 매주 학생 로그 샘플링
        ↓
[Production OK rate]
        ├─ ≥ Golden score - 10pt → 정상
        └─ < Golden score - 10pt → ⚠️ 분포 이동, Golden Set 갱신 필요
```

## 13. 통과 후 다음 단계

이 명세서 구현으로 평가 신뢰도가 확보되면:

3. **HyDE 효과 정량 측정**: HyDE on/off A/B 테스트

## 14. 자주 하는 실수 5가지

1. **Synthetic 점수만 보고 통과 판정** ← 우리가 빠진 함정
2. **Golden Set이 너무 작음 (10~20건)** — 통계적 신뢰도 부족, 최소 50건
3. **Golden Set을 자주 바꿈** — frozen 원칙 어김, 비교 불가능
4. **end-to-end만 측정** — retrieval/generation 분리 안 함
5. **LLM judge를 보정 없이 신뢰** — 사람 판단과 다를 수 있음
