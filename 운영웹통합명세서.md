## 1. 통합 명세서의 목적과 범위

### 1.1 무엇을 만드는가

두 가지를 한 시스템으로 만듭니다.

1. **메타데이터 v3 스키마** — RAG 검색·운영을 위한 표준 메타 구조
2. **docs 스타일 운영 웹** — 비개발자가 메타·콘텐츠를 직접 편집하고 인덱싱까지 트리거할 수 있는 웹 UI

핵심 차별점은 **docs 사이트와 운영 도구가 합쳐진 형태**입니다:

- 사용자(학생·교직원) 입장에서 보면 → **읽기 좋은 docs 사이트**
- 운영자 입장에서 보면 → **인라인 편집·메타 관리·인덱싱 트리거**가 가능한 도구

GitBook, Notion, Docusaurus 같은 docs 사이트의 좌측 트리 + 메인 콘텐츠 + 메타 패널 구조를 따르되, 모든 항목이 RAG 인덱싱 대상이며 메타데이터로 종속관계가 표현됩니다.

### 1.2 개발 단계 한정 (제외 사항)

이 명세서는 **개발 단계** 사용을 전제로 합니다. 다음은 의도적으로 제외:

- ❌ KPI 모니터링·대시보드 (운영 단계에서 추가)
- ❌ 운영 단계 보안 (OAuth SSO·Casbin RBAC·감사 로그)
- ❌ 학교 SSO 연동
- ❌ 데이터 백업·복구 정책
- ❌ 사용자 신고 큐 (운영 시 추가)

이 영역은 운영 단계 진입 시 별도 명세서로 다룹니다.

## 2. 우리 데이터의 종속관계 분석

운영 웹의 트리 구조를 설계하려면 먼저 **데이터의 자연스러운 계층**을 파악해야 합니다. 우리 corpus를 분석한 결과 컬렉션마다 다른 종속관계를 가집니다.

### 2.1 컬렉션별 계층 구조

| 컬렉션 | 자연스러운 계층 | 트리 표현 깊이 |
|---|---|---|
| **학칙_조항** | 학칙 > 장 > 조 > 항 | 4단계 |
| **학사일정** | 학사일정 > 학기 > 이벤트 | 3단계 |
| **강의평가** | 강의평가 > 분야 > 강의 > 섹션 | 4단계 |
| **FAQ** | FAQ > 카테고리 > 질문 | 3단계 |
| **시설_연락처** | 시설 > 건물·유형 > 항목 | 3단계 |
| **장학금** | 장학금 > 종류 > 항목 | 3단계 |
| **학사정보** | 학사정보 > 주제 > 절차 | 3단계 |
| **학과정보** | 학과 > 학과명 > 항목 | 3단계 |
| **교육과정** | 교육과정 > 학과 > 트랙 | 3단계 |

### 2.2 실제 데이터 예시로 본 계층

**학칙_조항** (4단계):

```
📚 학칙
└─ 📖 제3장 학사
   └─ 📄 제31조 (수료)
      ├─ 📋 1항: 매 학년 30학점 이상 취득해야 한다
      ├─ 📋 2항: 1학년 수료에 필요한 학점은…
      └─ 📋 3항: 단, 의학과는…
```

**강의평가** (4단계):

```
🎓 강의평가
└─ 📚 어학 (분야)
   └─ 📘 글로벌문화영어 (강의)
      ├─ 📝 교수님별 상세 특징
      ├─ 📝 시험 및 평가 방식
      ├─ 📝 과제 및 참여
      ├─ 📝 출결 관리
      ├─ 📝 난이도 및 학점
      ├─ 📝 장점
      ├─ 📝 단점 및 주의사항
      └─ 📝 추천 대상
```

**학사일정** (3단계):

```
📅 학사일정
└─ 📅 2026학년도 1학기
   ├─ 📆 수강바구니(1차) — 2026-01-15
   ├─ 📆 등록 — 2026-02-15~28
   ├─ 📆 입학식 — 2026-03-02
   └─ 📆 개강 — 2026-03-02
```

**FAQ** (3단계):

```
❓ FAQ
└─ 💬 수강신청/정정 (17개)
   ├─ ❓ 수강신청은 언제부터?
   ├─ ❓ 수강 정정 기간은?
   └─ ❓ 학점 초과 신청 가능한가요?
```

### 2.3 종속관계를 메타데이터로 표현하는 방법

평탄 dict 원칙을 지키면서 종속관계를 표현하려면:

| 방법 | 설명 | 예시 |
|---|---|---|
| `parent_doc_id` | 직계 부모 청크 ID (계층 1단계) | `parent_doc_id: "학칙_제31조"` |
| `path` (str, 슬래시 구분) | 전체 경로 한 문자열로 | `path: "학칙/제3장/제31조/1항"` |
| `breadcrumb` (list) | UI 표시용 경로 배열 | `["학칙", "제3장", "제31조", "1항"]` |
| `chunk_index` | 같은 부모 안의 순서 | `chunk_index: 0` |

**선택**: `parent_doc_id` + `path` + `breadcrumb` 세 개 모두 사용. 평탄 dict라 BM25 필터에도 안전, 트리 렌더링도 쉬움.

## 3. 메타데이터 v3 스키마 (통합)

### 3.1 3-Layer 구조

```
Layer 1: Core (모든 청크 필수)        → 임베딩 일부 포함, LLM 컨텍스트 일부 포함
Layer 2: Domain (컬렉션별 조건부)     → 임베딩 일부 포함
Layer 3: Operations (운영용)          → 임베딩·LLM 모두 제외
```

### 3.2 Layer 1 — Core (모든 청크 필수)

| 필드 | 타입 | 예시 | 용도 |
|---|---|---|---|
| `doc_id` | str (PK) | `"학칙_제31조_p1"` | 고유 ID |
| `parent_doc_id` | str | `"학칙_제31조"` | 직계 부모 청크 |
| `path` | str | `"학칙/제3장/제31조/1항"` | 슬래시 구분 경로 |
| `breadcrumb` | list[str] | `["학칙","제3장","제31조","1항"]` | UI 트리 |
| `schema_version` | str | `"v3"` | 마이그레이션 추적 |
| `source_collection` | str (enum) | `"학칙_조항"` | 라우팅 핵심 |
| `category` | str | `"학칙"` | 답변 인용용 |
| `subcategory` | str | `"학사"` | 보조 라우팅 |
| `title` | str | `"제31조 (수료)"` | 화면 표시 |
| `campus` | str (enum) | `"전체"` | 캠퍼스 필터 |
| `language` | str | `"ko"` | 다국어 확장 대비 |
| `chunk_index` | int | `0` | 같은 부모 내 순서 |
| `chunk_count` | int | `3` | 부모 아래 전체 청크 수 |
| `depth` | int | `4` | 트리 깊이 |

### 3.3 Layer 2 — Domain (컬렉션별 조건부)

각 컬렉션이 필요한 필드만 채웁니다. 다른 컬렉션에서는 None 또는 키 부재.

#### 학칙_조항

| 필드 | 타입 | 예시 |
|---|---|---|
| `chapter` | str | `"제3장"` |
| `chapter_title` | str | `"학사"` |
| `article_number` | str | `"제31조"` |
| `article_title` | str | `"수료"` |
| `paragraph` | int | `1` |

#### 학사일정

| 필드 | 타입 | 예시 |
|---|---|---|
| `start_date` | str (ISO) | `"2026-08-12"` |
| `end_date` | str (ISO) | `"2026-08-14"` |
| `semester` | str | `"2026-1"` |
| `event_type` | str | `"수강신청"` |

#### 강의평가

| 필드 | 타입 | 예시 |
|---|---|---|
| `lecture_id` | str | `"lecture_reviews_0"` |
| `lecture_title` | str | `"글로벌문화영어"` |
| `section` | str | `"시험 및 평가 방식"` |
| `subject_area` | str | `"어학"` |
| `is_required` | bool | `false` |

#### FAQ ★ (검색 보강 핵심)

| 필드 | 타입 | 예시 |
|---|---|---|
| `question_canonical` | str | `"교내장학금은 어떻게 신청하나요?"` |
| `question_variants` | list[str] | `["장학금 신청 어디서?", ...]` |
| `keywords` | list[str] | `["장학금", "신청", "학생포털"]` |
| `negative_examples` | list[str] | `["국가장학금 신청법"]` |

#### 시설_연락처

| 필드 | 타입 | 예시 |
|---|---|---|
| `phone` | str | `"031-740-7310"` |
| `building` | str | `"뉴밀레니엄센터"` |
| `floor` | str | `"4층"` |
| `facility_type` | str | `"학과사무실"` |

#### 장학금

| 필드 | 타입 | 예시 |
|---|---|---|
| `scholarship_type` | str | `"성적우수"` |
| `application_period_start` | str (ISO) | `"2026-02-01"` |
| `application_period_end` | str (ISO) | `"2026-02-28"` |
| `eligibility_grade` | float | `3.8` |

#### 교육과정·학과정보

| 필드 | 타입 | 예시 |
|---|---|---|
| `department` | str | `"간호학과"` |
| `low_confidence` | bool | `true` |

### 3.4 Layer 3 — Operations (운영용)

**핵심**: 이 레이어는 검색·답변 생성에 영향 없음. 오직 운영 추적 용도.

| 필드 | 타입 | 예시 | 용도 |
|---|---|---|---|
| `effective_start` | str (ISO) | `"2026-03-01"` | 정보 유효 시작일 |
| `effective_end` | str (ISO) | `null` | 만료일 (만료 후 검색 제외) |
| `created_at` | str (ISO+TZ) | `"2026-04-26T00:00:00+09:00"` | 원본 생성 시각 |
| `indexed_at` | str (ISO+TZ) | `"2026-04-26T10:00:00+09:00"` | 인덱싱 시각 |
| `last_verified_at` | str (ISO) | `"2026-04-15"` | 사람 검증 시점 |
| `version` | int | `1` | 콘텐츠 버전 |
| `owner` | str | `"학생지원팀"` | 담당 부서 |
| `confidence` | str (enum) | `"high"` | 신뢰도 |

### 3.5 Enum 정의

```python
VALID_SOURCE_COLLECTIONS = [
    "강의평가", "학칙_조항", "학사정보", "FAQ",
    "시설_연락처", "학사일정", "장학금", "교육과정",
    "학과정보", "기타"
]

VALID_CAMPUSES = ["성남", "의정부", "대전", "전체"]

VALID_CONFIDENCE = ["high", "medium", "low"]

VALID_LANGUAGES = ["ko", "en"]
```

### 3.6 임베딩·LLM 노출 전략 (LlamaIndex MetadataMode 패턴)

같은 메타라도 **검색용·LLM용·운영용**을 분리합니다.

```python
# 임베딩에 포함되는 필드 (검색 정확도 향상)
EMBED_INCLUDE_FIELDS = [
    "title", "category", "subcategory", "campus",
    "section", "keywords", "question_canonical", "question_variants",
    "article_number", "article_title", "lecture_title",
]

# LLM 답변 생성 시 컨텍스트에 포함되는 필드
LLM_INCLUDE_FIELDS = [
    "title", "category", "campus", "breadcrumb",
    "article_number", "article_title", "section",
    "start_date", "end_date", "phone",
]

# 운영 메타 (검색·답변 모두 제외)
OPERATIONS_FIELDS = [
    "effective_start", "effective_end", "created_at", "indexed_at",
    "last_verified_at", "version", "owner",
    "confidence", "schema_version",
]
```

## 5. 쿼리 보강: HyDE (확정)

### 5.1 HyDE 개요

HyDE (Hypothetical Document Embeddings)는 **사용자 쿼리 → 가상의 답변 문서 → 그 문서를 임베딩하여 검색**하는 방식입니다. 사용자 쿼리는 짧고 추상적이지만, 답변 문서는 구체적이라 임베딩 매칭이 좋아집니다.

```
사용자 쿼리: "졸업학점은?"
    ↓
LLM (Solar Pro)이 가상 답변 생성:
    "졸업을 위해서는 최소 130학점 이상을 취득해야 합니다.
     전공필수·전공선택·교양필수·교양선택 학점 비율이 정해져 있으며..."
    ↓
이 가상 답변을 임베딩 → Qdrant 검색
    ↓
실제 정답 청크 매칭 정확도 ↑
```

### 5.2 우리 시스템에서의 흐름

```python
# 의사코드 (실제 구현 시 src/retrieval/hyde.py)
async def search_with_hyde(query: str, metadata_filter: dict | None = None):
    # 1. HyDE로 가상 답변 생성
    hyde_doc = await solar_pro.generate(
        f"다음 질문에 대한 답변을 1~2문장으로 작성하세요: {query}",
        max_tokens=200,
        temperature=0.3,
    )

    # 2. 가상 답변을 임베딩
    hyde_embedding = await solar_embed_query(hyde_doc)

    # 3. Qdrant 검색 (메타 필터 함께)
    dense_results = await qdrant.search(hyde_embedding, top_k=30, filter=metadata_filter)

    # 4. BM25는 원본 쿼리로 (HyDE 적용하면 키워드 매칭 망가짐)
    sparse_results = await bm25.search(query, top_k=30, filter=metadata_filter)

    # 5. RRF 융합
    return rrf_fusion(dense_results, sparse_results, k=60)
```

### 5.3 HyDE 적용 시 주의사항

- **BM25에는 HyDE 적용 X** — 가상 답변에는 학생이 쓸 키워드가 없음
- **HyDE 답변은 캐싱** (Redis, 동일 쿼리 재호출 방지)
- **temperature 0.3~0.5** — 너무 낮으면 다양성 부족, 너무 높으면 환각
- **max_tokens 200 정도** — 너무 길면 임베딩에 노이즈 추가

### 5.4 HyDE 프롬프트 (한국어 최적화)

```
프롬프트 템플릿:
"당신은 을지대학교 학사 정보 전문가입니다.
다음 질문에 대한 답변을 1~2문장으로 간결하게 작성하세요.
답변은 학칙, 학사정보, 또는 FAQ에 있을 만한 내용으로 작성하세요.

질문: {query}

답변:"
```

이 프롬프트는 검색용이므로 정확하지 않아도 됩니다. 임베딩이 정답 청크 근처로 가도록 유도하는 것이 목표.

### 6.3 좌측 트리 (Tree Sidebar)

종속관계를 자연스럽게 표현하는 핵심 컴포넌트.

#### 트리 구성 규칙

```
{컬렉션 아이콘} {컬렉션명}
└─ {breadcrumb의 각 단계가 노드로}
   └─ ...
      └─ {말단 청크 = doc_id}
```

#### 트리 노드 종류

| 노드 타입 | 의미 | 클릭 동작 |
|---|---|---|
| 📚 컬렉션 (depth=1) | 10개 컬렉션 | 컬렉션 메인 페이지 |
| 📖 그룹 (depth=2~3) | 장·학기·분야·서브카테고리 | 자식 목록 표시 |
| 📄 문서 (depth=중간) | 조·강의·이벤트·질문 | 콘텐츠 + 자식 표시 |
| 📋 청크 (depth=최대) | 항·섹션·청크 | 콘텐츠 + 메타 |

#### 트리 데이터 만드는 법 (백엔드)

```python
# /api/tree?collection=학칙_조항
def build_tree(chunks: list[ChunkRecord]) -> TreeNode:
    """평탄 청크 리스트를 path로 트리 구축"""
    root = {"name": "학칙", "children": {}}
    for chunk in chunks:
        node = root
        for part in chunk.breadcrumb:
            if part not in node["children"]:
                node["children"][part] = {"name": part, "children": {}, "doc_id": None}
            node = node["children"][part]
        node["doc_id"] = chunk.doc_id  # 말단 청크
    return root
```

#### 트리에 보일 시각 신호

| 신호 | 의미 |
|---|---|
| 🟢 색상 | Indexed (정상) |
| 🟡 색상 | Published (인덱싱 대기) |
| ⚪ 색상 | Draft (작성 중) |
| 🔴 색상 | Expired (만료) |
| ⚠️ 아이콘 | last_verified_at이 6개월 지남 |
| 🔒 아이콘 | low_confidence=true (정보 빈약) |

## 7. 컬렉션별 화면 차이

각 컬렉션의 트리 깊이·메타 필드가 달라서 화면도 일부 달라집니다.

### 7.1 학칙_조항 (4단계 트리, 가장 구조적)

- 좌측 트리 4단계: 학칙 > 장 > 조 > 항
- 자동 생성: chapter·article_number·paragraph 정규식 파싱
- 검토 권한 필수: `version > 1`인 변경은 검토자 승인
- "관련 조항" 섹션: 같은 장 안의 인접 조

### 7.2 강의평가 (4단계 + 마크다운 헤더)

- 트리: 강의평가 > 분야 > 강의 > 섹션 (## 헤더)
- 강의 페이지 전체를 보면 9~12개 섹션이 펼쳐짐
- 메타 패널에 `lecture_id` 그룹화로 같은 강의 다른 섹션 점프
- 학생 자동 입력 위주이므로 운영자 수동 편집 적음 — `confidence: medium` 기본

### 7.3 학사일정 (3단계 + 시간축)

- 트리: 학사일정 > 학기 > 이벤트
- 추가 뷰: **캘린더 뷰** (FullCalendar 라이브러리 사용)
- 자동 검증: `start_date <= end_date`
- 자동 만료: `effective_end < today`인 청크는 좌측 트리에서 흐릿하게

### 7.4 FAQ ★ (자동 보조 가장 적극)

- 트리: FAQ > 카테고리 > 질문
- 신규 작성 시 3-step 마법사:
  1. 표준 질문 + 답변 입력
  2. Solar Pro로 variants 5개 자동 생성 → 사용자 검수
  3. keywords·negative_examples 자동 추출 → 사용자 검수
- 메타 패널에 variants 목록 인라인 편집

### 7.5 시설_연락처 (3단계, 정적 정보)

- 트리: 시설 > 건물·유형 > 항목
- 전화번호 정규식 검증 (`^0\d{1,2}-\d{3,4}-\d{4}$`)
- 지도 보기 옵션 (선택)

### 7.6 장학금 (3단계 + 신청기간)

- 트리: 장학금 > 종류 > 항목
- 추가 뷰: **신청 기간 타임라인** (학사일정과 비슷)
- `application_period_end`가 지난 항목은 "신청 마감" 배지

### 7.7 학사정보·학과정보·교육과정·기타 (3단계 일반)

- 표준 3단계 트리
- low_confidence=true 항목은 ⚠️ 표시 + 답변 시 경고

## 9. 데이터베이스 스키마 (PostgreSQL)

```sql
-- 청크 메타·콘텐츠
CREATE TABLE chunks (
    doc_id VARCHAR PRIMARY KEY,
    parent_doc_id VARCHAR,
    path VARCHAR NOT NULL,
    schema_version VARCHAR DEFAULT 'v3',
    source_collection VARCHAR NOT NULL,
    metadata JSONB NOT NULL,           -- v3 평탄 dict 전체
    contents TEXT NOT NULL,             -- 임베딩 본문 (prefix 포함)
    raw_content TEXT,                   -- 원본 본문
    status VARCHAR NOT NULL DEFAULT 'Draft',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_chunks_status ON chunks(status);
CREATE INDEX idx_chunks_collection ON chunks(source_collection);
CREATE INDEX idx_chunks_parent ON chunks(parent_doc_id);
CREATE INDEX idx_chunks_path ON chunks(path);
CREATE INDEX idx_chunks_metadata_gin ON chunks USING gin(metadata);

-- 변경 이력
CREATE TABLE chunk_history (
    id SERIAL PRIMARY KEY,
    doc_id VARCHAR REFERENCES chunks(doc_id) ON DELETE CASCADE,
    version INT NOT NULL,
    changed_at TIMESTAMPTZ DEFAULT NOW(),
    diff JSONB NOT NULL                 -- {field: [before, after]}
);

CREATE INDEX idx_chunk_history_doc ON chunk_history(doc_id, version);

-- 인덱싱 작업
CREATE TABLE indexing_jobs (
    id SERIAL PRIMARY KEY,
    job_type VARCHAR,                   -- incremental/full
    status VARCHAR DEFAULT 'queued',    -- queued/running/success/failed
    started_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ,
    chunks_total INT,
    chunks_processed INT DEFAULT 0,
    error_message TEXT
);
```

## 13. Solar Pro 자동 보조 시점 정리

운영 웹의 가치를 극대화하는 자동 보조 기능들. 모두 **사람 검수가 마지막 단계**.

| 기능 | 발동 시점 | 출력 |
|---|---|---|
| `question_variants` 5개 생성 | FAQ 작성 중 [✨] 버튼 | 5개 변형 질문 |
| `keywords` 추출 | FAQ 답변 입력 후 자동 | 키워드 5개 |
| `negative_examples` 제안 | FAQ 답변 입력 후 자동 | 비슷하지만 답이 다른 질문 |
| 학칙 chapter·article 파싱 | 학칙 작성 시 자동 | 정규식 + LLM 검증 |
| 강의평가 ## 헤더 자동 분할 | CSV 업로드 시 | 청크 자동 생성 |
| answer_short 요약 | FAQ 답변 작성 후 | 1문장 요약 |
| 학사일정 날짜 정규화 | 입력 시 | ISO 8601 변환 |
| HyDE 가상 답변 | 검색 미리보기 시 | 검색 보강 |

## 14. 잠재적 리스크와 대응

| 리스크 | 대응 |
|---|---|
| 트리가 너무 깊어 UX 나쁨 | react-arborist 가상화 + lazy loading |
| 큰 corpus(2,400+ 청크) 트리 렌더링 느림 | 컬렉션별 분리 트리, 검색으로 직접 점프 |
| HyDE 호출 비용 증가 | Redis 시맨틱 캐싱 (동일 쿼리 재호출 방지) |
| Solar Pro 자동 생성 품질 낮음 | 사용자 검수 단계 필수, 거부 가능 |
| 동시 편집 충돌 | `version` 필드 + 낙관적 락, 충돌 시 재로딩 요청 |
| TipTap 마크다운 호환성 | 보기 모드는 `react-markdown`, 편집은 TipTap (분리) |
| 인덱싱 중 작업 큐 안 비워짐 | Celery 진행률 모니터링, 실패 시 재시도 큐 |
